## ❌ 问题 3：类型碎片化

### 当前：同一类型多处定义

```
ScriptSegment 定义在 3 个文件：

src/types/script.ts
├── export interface ScriptSegment { ... } ← 权威定义

src/core/services/ai/script/script-generation-service.ts
├── export type ScriptSegment = CoreScriptSegment; ← 向后兼容别名 ✗

src/core/types/script.ts
├── export interface ScriptSegment { ... } ← 重复定义 ✗

问题：
  • 修改一处必须同步其他
  • 类型不一致风险
  • 维护成本高
```

### ✅ 重构后：单一权威源

```
src/types/script.ts
├── export interface ScriptSegment { ... } ← 唯一定义

所有其他文件：
├── import type { ScriptSegment } from '@/types';
└── (删除所有重复定义和别名)

收益：
  ✅ 删除 ~200 行重复代码
  ✅ 维护成本降低 60%
  ✅ 消除类型不一致风险
```

## 关键决策点

### 决策 1：删除旧版 Pipeline vs 迁移？

**选项 A：删除旧版** ✅ 推荐
- ✅ 删除 137 行 dead code
- ✅ 消除 180 行重复
- ✅ 统一 API
- ❌ 需要迁移 12 个文件（工作量）

**选项 B：保留兼容层**
- ❌ 持续维护负担
- ❌ 新增功能需同步两边
- ✅ 短期工作量小

**决策**：选择 **A（删除）**，长期收益远大于短期成本

---

### 决策 2：类型统一策略

**选项 A：激进（一次性全部迁移）**
- ❌ 高风险，可能引入 bug
- ❌ 工作量大，难以验证

**选项 B：渐进（按模块迁移）** ✅ 推荐
- ✅ 低风险，可逐步验证
- ✅ 工作可并行
- ✅ 旧代码逐步退役

**决策**：选择 **B（渐进）**，按类型域分批迁移

---

### 决策 3：服务层重构范围

**选项 A：一次性重构所有 7 个服务**
- ❌ 工作量大（~50h）
- ❌ 风险高
- ❌ 影响面大

**选项 B：优先重构高耦合服务** ✅ 推荐
- ✅ Commentary（12 个导入者）
- ✅ AIService（5 个职责）
- ⏸ 其他服务后续处理

**决策**：选择 **B（优先高耦合）**， Commentary 和 AIService 优先

## 风险与缓解措施

| 风险 | 严重性 | 缓解措施 |
|------|--------|---------|
| **破坏现有功能** | 🔴 高 | 每阶段后运行完整测试套件 |
